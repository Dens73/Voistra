# TODO — Desktop VoIP дипломный проект

## 1. Цель проекта

Разработать desktop-приложение клиент-серверной VoIP-связи с поддержкой:
- авторизации пользователей
- серверов и каналов
- текстовых и голосовых каналов
- многопользовательских аудиоканалов до 10 участников
- push-to-talk и voice activation
- mute / deafen / speaking state
- ролей и прав доступа
- приватных каналов с паролем
- демонстрации экрана без записи
- передачи системного звука при демонстрации экрана
- базового шумоподавления на клиенте
- индикации качества сети
- логирования и административных функций

## 3. Архитектурные решения

- [ ] Desktop-only приложение
- [ ] Клиент и сервер работают через интернет
- [ ] Голосовые каналы реализуются через SFU
- [ ] Screen sharing передаётся как отдельный медиапоток
- [ ] Шумоподавление выполняется на клиенте
- [ ] Базовая архитектура: один backend + один mediasoup worker/набор workers + PostgreSQL + Redis + coturn
- [ ] Целевая нагрузка: до 10 участников в одном голосовом канале
- [ ] Поддержка большого количества серверов и комнат на уровне модели данных
- [ ] Горизонтальную масштабируемость считать необязательной для MVP, но заложить архитектурно возможность расширения через Redis/pub-sub и разделение signaling/media слоёв

## 4. Важные ограничения и договорённости

- [ ] Видеозвонки между участниками не реализуются
- [ ] Запись звонков и записи демонстрации экрана не реализуются
- [ ] Гостевой доступ не нужен
- [ ] Аудит и журналов действий должны быть
- [ ] Качество звука при плохой сети не является главным приоритетом
- [ ] Автоматический reconnect должен быть настраиваемой опцией `on/off`

## 5. Важные технические примечания

### NAT traversal
- [ ] Реализовать прохождение NAT через ICE + STUN + TURN
- [ ] Использовать coturn как обязательный компонент для удалённой работы через интернет
- [ ] Не формулировать требование как «идеальная работа за NAT без проблем»; корректная цель: «максимально надёжное установление соединения в типовых сетевых условиях через ICE/STUN/TURN»

### E2E-шифрование медиа
- [ ] Не включать E2E-шифрование медиа в базовый MVP
- [ ] Зафиксировать E2E как advanced / исследовательскую функцию второго этапа
- [ ] В базовой версии использовать стандартную защиту WebRTC-трафика
- [ ] Отдельно изучить совместимость E2E с архитектурой SFU + mediasoup
